# AuditGPT Output Contract

This document defines what AuditGPT's verification badge does and does not
guarantee about audit output. It is the honest scope the badge points to.
The checks below are implemented in `src/lib/eval/contract.ts` and run by
`scripts/eval.ts` (`bun run eval`) on every push and pull request.

The badge cites only the **hard checks**. It does not claim correctness it
cannot measure.

---

## Contract hardness

Three tiers, ordered by how strong the guarantee is.

### 1. Hard checks — enforced, block CI

A failure exits non-zero and fails the `contract-eval` job, which blocks
merge when the check is marked required. These are deterministic string
facts with no interpretation.

| Check | Rule | Method |
|---|---|---|
| Banned phrases | No `great`, `exciting`, `however`, `delusion` (word-boundary), or `let's be honest`, `the reality is`, `founder probably believes`, `learn more` (substring) | Regex/substring over every scanned string field and array element |
| Sentence cap | Maximum 2 sentences per field | Sentence split on `.!?`, abbreviations protected, count > 2 fails |
| Insufficient-data coverage | Fields known to lack input data must contain the literal string `insufficient data` | Per-fixture expectation list; each expected field is asserted present |

Scanned fields: `verdict`, `verdict_header`, `company_info`, and every
element of `report_card`, `red_flags`, `assumptions_to_test`,
`website_fixes`, `services_to_hire`, `action_plan`.

Why these exist: banned phrases are the surface signals of generated
filler; the 2-sentence cap forces compression that filler does not survive;
the insufficient-data requirement is the anti-hallucination core — output
must say it has no data rather than invent a number.

### 2. Advisory check — flagged, never blocks

| Check | What it does | Why advisory |
|---|---|---|
| Untraced numbers | Flags numbers in output that are absent from the fixture's allowed set (input values + hardcoded benchmarks + the 0–100 score scale) | It cannot prove a number was invented. It produces false positives — for example a generated score reappearing inside `verdict_header` — so it routes to human review instead of failing the build |

A clean run can still report advisory flags. On the current recorded set,
one advisory flag is expected (a score value echoed in `verdict_header`).

### 3. Not machine-checkable — declared, not enforced

These appear in the prompt's banned list but cannot be pattern-matched
reliably, so the harness does not test them and the badge does not cover
them:

- **Metaphor and simile.** No regex distinguishes a metaphor from literal
  comparison without false positives. Excluded from enforcement.
- **"Every sentence must contain a fact."** Whether a sentence carries a
  number, proper name, or benchmark is a judgment the harness does not make.
- **"If a sentence feels inspiring, rewrite it."** Subjective; not testable.

A reviewer remains responsible for these. The badge claim is scoped to the
hard checks precisely so it never implies coverage the harness lacks.

## Evaluation modes

- **Recorded mode** (default, CI): runs the contract against stored
  input/output fixtures in `src/lib/eval/fixtures.ts`. Deterministic, no
  network, no secrets, no model cost. This is what the badge cites.
- **Live mode** (`--live`, pre-release): regenerates fixture outputs from
  the real audit path and re-runs the identical contract. Detects model
  drift before a provider or prompt change ships.

---

## Scope and limitations

- **Coverage is over the recorded fixture set, not all possible audits.**
  A passing run proves the contract held on those specific outputs. It does
  not prove the contract holds universally. Coverage strength grows only by
  adding fixtures.
  
  **⚠️ Current fixture set is thin (2 cases).** Before showcasing the badge
  to paying customers, expand `src/lib/eval/fixtures.ts` with real audit
  outputs — especially cases that previously violated the contract (slipped
  banned phrases, missing insufficient-data markers). Those regressions are
  what the harness exists to catch.

- **Model dependency.** Output is generated by `z-ai-web-dev-sdk` v0.0.18.
  The eval measures that vendor's behavior. A second provider must be run
  through the same contract before its output inherits the badge.

- **The 2-sentence and banned-phrase checks read from one source.**
  `BANNED_WORDS` and `BANNED_PHRASES_MULTIWORD` in `contract.ts` are
  imported into `audit-context.ts` and interpolated into the preamble,
  so the test and the prompt cannot drift apart.

The badge must state the measured scope, not "never hallucinates" (which the

## Single-source-of-truth implementation

The banned-phrase lists are defined once in `src/lib/eval/contract.ts`:

```typescript
export const BANNED_WORDS = ['great', 'exciting', 'however', 'delusion'];
export const BANNED_PHRASES_MULTIWORD = ["let's be honest", "the reality is", ...];
```

This is imported and used in two places:

1. **Eval harness** (`scripts/eval.ts`): runs `checkBannedPhrases()` against outputs
2. **Audit prompt** (`src/lib/audit-context.ts`): interpolates into `ANTI_SLOP_PREAMBLE`

When you edit `BANNED_WORDS` or `BANNED_PHRASES_MULTIWORD` in contract.ts,
both the hardness test and the LLM prompt update together. No manual sync needed.
